Proceso de migración paso a paso

⚠️ Antes de empezar

Hacer la migración un domingo en la noche cuando el tráfico es mínimo. El proceso completo toma ~30-45 minutos y la app estará intermitente durante ~5 minutos del paso de DNS.

Fase 1 — Crear el Snapshot

~10 minutos · Sin downtime

  • 1
    Abrir el droplet en DigitalOcean
    Dashboard → Droplets → ubuntu-s-2vcpu-4gb-sfo2-01
  • 2
    Ir a la pestaña "Snapshots"
    Menú izquierdo del droplet → Snapshots
  • 3
    Crear snapshot
    Clic en "Take Snapshot" → nombre: m7app-sfo2-2025-DD
    ℹ️ La app sigue corriendo mientras se crea el snapshot. No habrá caída en este paso.
  • 4
    Esperar a que termine
    Tarda entre 5 y 15 minutos dependiendo del tamaño del disco

Fase 2 — Crear el Nuevo Droplet en NYC1

~5 minutos · Sin downtime

  • 1
    Ir a Images → Snapshots
    DigitalOcean → Manage → Images → Snapshots
  • 2
    Crear droplet desde el snapshot
    Buscar el snapshot m7app-sfo2 → clic en los "..." → "Create Droplet"
  • 3
    Configurar el nuevo droplet
    Región: Nueva York 1 (NYC1)
    Plan: 2 vCPU / 4 GB RAM / $24/mo
    Auth: SSH Key (la misma existente)
    Hostname: ubuntu-s-2vcpu-4gb-nyc1-01
  • 4
    Anotar la nueva IP
    DigitalOcean asigna una IP nueva. Guardarla — la necesitas en los siguientes pasos.

Fase 3 — Actualizar Coolify

~5 minutos

  • 1
    Abrir Coolify → Settings → Servers
    En el panel lateral de Coolify
  • 2
    Seleccionar el servidor actual
    El que apunta a la IP 138.68.245.236 (SFO2)
  • 3
    Cambiar la IP
    Campo "IP Address" → reemplazar 138.68.245.236 por la nueva IP de NYC1
  • 4
    Verificar conexión SSH
    Coolify → el servidor → botón "Test Connection" o "Validate". Debe aparecer en verde.
  • 5
    Hacer un deploy de prueba
    En cualquiera de las apps (frontend o backend) → botón "Deploy" → verificar que completa sin errores

Fase 4 — Actualizar DNS

~2 minutos de trabajo · 5-30 min de propagación

⚠️ Este es el único paso con impacto en usuarios. Habrá ~5 minutos donde algunas peticiones van al servidor viejo y otras al nuevo. Hacerlo de noche minimiza el impacto.
  • 1
    Entrar al proveedor de dominio (Cloudflare / GoDaddy / otro)
    Donde esté registrado el dominio de OrbitM7
  • 2
    Buscar el registro tipo "A" del dominio
    DNS → Records → buscar el registro A que apunta a 138.68.245.236
  • 3
    Cambiar la IP destino
    Reemplazar la IP vieja por la nueva IP del droplet NYC1
    💡 Si usas Cloudflare: bajar el TTL a 1 minuto antes de cambiar la IP, esperar 5 min, cambiar la IP, esperar 5 min más, luego volver el TTL a "Auto". Esto hace la transición casi instantánea.
  • 4
    Verificar propagación
    Abrir la app en el navegador. Si abre y funciona normalmente, el DNS propagó correctamente.

Fase 5 — Destruir el Droplet Viejo

Solo cuando todo funcione · Acción irreversible

🚨 No destruir el droplet viejo hasta confirmar que la app funciona correctamente en el nuevo — incluyendo login, carga de datos, uploads a Drive, y un deploy completo desde Coolify.
  • 1
    Confirmar que todo funciona en NYC1
    Probar login, rutas, documentos, PDFs, Cumplidos Drive, y que Coolify puede hacer deploy al nuevo servidor
  • 2
    Esperar 24 horas (opcional, por seguridad)
    Si el lunes todo corre normal, el martes se puede destruir el droplet SFO2
  • !
    Destruir el droplet SFO2
    DigitalOcean → Droplets → ubuntu-s-2vcpu-4gb-sfo2-01 → Destroy → confirmar
Optimizaciones en Coolify (hacer antes/durante)

Configurar Zero Downtime Deployment

Mantiene el contenedor viejo activo mientras el nuevo levanta. Los usuarios no notan el deploy.

Coolify → seleccionar la app (backend o frontend) → pestaña General
Buscar la opción "Zero Downtime Deployment" y activar el toggle
En la sección Health Check, configurar:
Path: /api/health · Interval: 10s · Timeout: 30s · Retries: 3
Guardar y hacer un deploy de prueba para verificar que el nuevo contenedor pasa el health check
Endpoint a agregar en backend/index.ts (si no existe)
// Antes de todas las rutas app.get('/api/health', (_req, res) => res.json({ ok: true }));

Configurar Backend para usar imagen GHCR

En lugar de que Coolify construya el backend en el servidor, lo jala desde GitHub Actions. El deploy pasa de ~8 min a ~30 segundos.

Coolify → la aplicación del backend → pestaña Source o General
Cambiar tipo de fuente de "Build from Source" a "Docker Image"
En el campo imagen poner:
ghcr.io/oesantama/m7-backend:latest
El workflow build-backend.yml ya está creado — el primer push a backend/ construirá la imagen en GitHub Actions y la subirá a GHCR
Guardar. A partir de ahora el servidor solo hace docker pull — sin build local

Configurar Variables de Entorno en Coolify

Verificar que todas las ENV del backend están en Coolify y no hardcodeadas en el Dockerfile.

Coolify → backend app → pestaña Environment Variables
Confirmar que existen: DATABASE_URL, JWT_SECRET, NODE_ENV=production, TZ=America/Bogota, y cualquier clave de servicios externos
Si falta alguna, agregarla aquí — nunca en el Dockerfile ni en el repositorio

🔄 Plan de Rollback (si algo falla)

El droplet SFO2 no se destruye hasta el día siguiente. Si algo sale mal durante la migración:

  • 1
    Revertir el DNS
    Cambiar el registro A de vuelta a 138.68.245.236 (IP vieja SFO2) — tarda ~5 min en propagar
  • 2
    Revertir Coolify
    Settings → Servers → volver la IP al valor original 138.68.245.236
  • 3
    La app vuelve a funcionar desde SFO2
    Como si nada hubiera pasado. El droplet NYC1 queda activo unos días mientras se investiga el problema
